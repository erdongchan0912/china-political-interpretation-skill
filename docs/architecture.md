### 工程化系统拆解路径
建议将整个开发过程分为四个阶段，每次只让 Vibe Coding 软件完成一个阶段的核心代码：

#### Phase 1: MVP 极简分析流 (核心跑通)
**工程目标**：不考虑任何历史对比、外部检索和自我进化，只把“单篇文档的结构化和要素提取”做出来。

- **输入 (Input)**：支持纯文本或单一的 PDF 上传。
- **处理 (Process)**：使用现成的文档解析库（如 PyMuPDF 或直接调用大模型视觉接口）将 PDF 转为文本。写一段固定的 Prompt，让大模型提取：核心政策目标、涉及部门、时间节点。
- **输出 (Output)**：将提取结果输出为标准的 JSON 格式，并用最简单的网页前端（如 Streamlit 或基础的 React/Vue 页面）渲染出来。
- **交付标志**：你能稳定地上传一篇政策，网页能稳定地吐出结构化的分析卡片。

#### Phase 2: RAG 与基础纵横对比 (引入历史维度)
**工程目标**：在 Phase 1 稳定的基础上，建立基础的本地知识库。

- **数据库搭建**：引入向量数据库（如 ChromaDB 或 Pinecone）。你需要手动上传一批过去几年的重要政策作为“冷启动数据”。
- **检索逻辑**：上传新文档后，系统先去向量数据库里检索出最相似的 3 篇历史文档。
- **Prompt 升级**：将新文档和检索出的历史文档一起喂给大模型，要求它执行“新增、删减、词汇平替”的比对分析。

#### Phase 3: 外部触角与高阶防错 (API 聚合)
**工程目标**：接入外部能力，实现异常预警和更精准的清洗。

- **OSINT 集成**：引入外部搜索引擎 API（如 SerpApi 或 Bing Search API）。当大模型在分析会议出席名单并发现“重要人物缺席”时，触发一段 Python 脚本，调用搜索 API 查验该人物近三天的公开行程。
- **日历算法**：引入中国工作日历的 Python 库（如 `chinesecalendar`），编写“政策时滞”的真实工作日计算逻辑。

#### Phase 4: 自动化与反馈闭环 (系统终态)
**工程目标**：实现系统的自我更新和强化学习。

- **定时任务 (Cron Jobs)**：编写爬虫或订阅 RSS，让系统每周自动抓取中国政府网的新文件，向量化后存入数据库。
- **数据回流**：在前端页面加上“点赞/报错”按钮。将用户的纠错数据存入关系型数据库（如 PostgreSQL），定期作为微调大模型（SFT）的训练集。

### 💡 工程边界信息与未曾关注的微小细节
在真正开始用代码实现时，以下几个工程层面的微小细节和物理限制，是你必须提前规划的，否则项目会在中途直接卡死：

- **上下文窗口塌陷与 API 成本失控 (Context Limit & Cost)**：
中国政策文件通常极长（动辄几万字）。在 Phase 2 中，如果你把 1 篇新政策加上 5 篇历史政策原文全部塞进大模型的上下文窗口，不仅很容易触发 Token 数量上限，还会引发大模型的“中间注意力丢失（Lost in the Middle）”问题，导致它忽略文档中间的关键政治信号。工程上必须设计极度精细的 **Chunking（文本分块）** 和 **reranking（重排）** 机制。
- **反爬虫机制与外部 API 的脆弱性 (Anti-crawler Walls)**：
在设想中，系统会自动抓取政府网站更新词库或核实行程。但在工程现实中，国内的政务网站有极度严格的防火墙和反爬虫策略（如动态 IP 封禁、复杂的验证码）。过度依赖实时爬虫会让你的工具极不稳定。可靠的工程替代方案是：购买成熟的政务数据商的 API 接口，或者退一步，采用半自动的导入模式。
- **Vibe Coding 的“文件覆写”陷阱 (File Overwriting Trap)**：
使用 AI 辅助编程时，如果你把所有逻辑（前端渲染、文档解析、大模型调用、数据库读写）都写在一个或两个巨大的文件里，当你要求 AI “加一个缺席人物查验功能”时，它极有可能在修改文件时把你之前写好的“文档解析逻辑”给删减或改坏。因此，**工程架构必须高度模块化（微服务或严格的函数分离）**。
